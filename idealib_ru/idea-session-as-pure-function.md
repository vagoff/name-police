## Формулировка
LLM-рабочую сессию нужно трактовать как чистую функцию: явные входы (загруженные юниты знания + текущая задача), явные выходы (результат + кандидаты новых идей). Состояние не должно накапливаться внутри сессии.

## Обоснование
Текущая модель по умолчанию — «диалог = stateful контекстное окно» — архитектурно дефектна для длинных рабочих сессий. Состояние накапливается имплицитно, соотношение сигнал/шум монотонно падает, сессия не воспроизводима и не компонуема.

Сессия как чистая функция имеет противоположные свойства: воспроизводима (те же входы → то же качество), компонуема (выходы одной сессии могут быть входами следующей), и её область явна (знаешь что загрузил, знаешь что произвёл).

Handoff-промпт в начале новой сессии — не костыль, а сигнатура функции. Его написание заставляет явно сформулировать какой контекст действительно важен.

Связь с → knowledge-unit-store: юниты — это загружаемые входы. Связь с → dialogue-mining: mining производит выходы. Связь с → context-degradation-as-signal: деградация маркирует естественную точку завершения функции.

Граница применимости: дисциплина чистой сессии требует внешнего хранилища идей. Без idealib входы для следующей сессии не существуют — и возвращаешься к stateful-накоплению.

## Примеры
**Хороший старт сессии:** «Контекст: [3 релевантных юнита вставлены]. Задача: спроектировать attention-window entry point для Tulip. Ожидаемый выход: решение + любые новые идеи для idealib.»

**Плохой паттерн сессии:** один тред диалога растянутый на несколько дней, тем и задач — балласт контекста растёт, релевантность старых поворотов стремится к нулю, деградация гарантирована.

## Следствия
- Длина сессии становится дизайнерским решением, а не случайностью «пока не устал».
- Качество сессии измеримо: сколько юнитов она произвела? Был ли выход достаточно чистым чтобы вставить его как вход следующей сессии?
- Idealib — это не просто PKM, это межсессионное хранилище состояния делающее чистые сессии возможными. Без него чистота остаётся только стремлением.
