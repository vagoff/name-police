## Формулировка
Git-коммит должен содержать ровно одну семантическую операцию: либо переименование, либо изменение содержимого — но не то и другое одновременно. Смешивание уничтожает историю которую git способен сохранить.

## Обоснование
Определение переименования в git эвристическое: оно сопоставляет удалённый файл с новым по схожести содержимого. Если переименование и изменение содержимого происходят в одном коммите, порог схожести может не быть достигнут и git запишет delete+create вместо rename. `git log --follow`, `git blame` и читаемость диффов — всё это зависит от корректного срабатывания этого определения.

Тот же принцип применим шире переименований: коммит рефакторинга (реструктуризация без изменения поведения) не должен смешиваться с исправлением бага. У каждой операции своя семантика — смешивание делает обе нечитаемыми при ревью и при археологии истории.

Граница применимости: применимо когда читаемость истории важна. Для одноразовых веток или сольного исследования стоимость лишних коммитов может превысить выгоду.

## Примеры
**Переименование + расширение:** `idea-xxx1.md` → `idea-xxx.md` (коммит 1, только rename), затем перезапись расширенным содержимым из `idea-xxx2.md` (коммит 2, только content). Git отслеживает файл как непрерывно эволюционирующий.

**Рефакторинг кода:** выделить функцию (коммит 1), затем починить баг внутри неё (коммит 2). При смешивании в один коммит `git blame` на строке с исправлением показывает автора/дату рефакторинга, а не исправления.

## Следствия
- `git log --follow <file>` работает сквозь переименования только если переименование было изолировано.
- Ревью изменений содержимого чистое: в диффе нет шума от переименования.
- Принцип обобщается: одна семантическая операция на коммит — это инвариант; rename vs content — лишь наиболее часто нарушаемый его случай.
