## Формулировка
В языке или API должен быть ровно один символ обозначающий присваивание — и он должен использоваться во всех контекстах где происходит присваивание. Контекстно-зависимые синонимы создают ментальную нагрузку без добавления информации.

## Обоснование
Если в одном месте языка используется `=`, в другом `:`, в третьем `=>` для семантически одинакового действия "имя получает значение" — пользователь вынужден помнить правила для каждого контекста. Это arbitrary complexity — сложность без смысловой причины.

Проверка: если объяснение правила использования символа содержит слово "в этом контексте" — это признак что символов больше чем нужно.

Граница применимости: разные символы оправданы когда они несут разную семантику (`:` как разделитель ключа и значения в литерале словаря — не присваивание, а структура). Речь о случаях когда семантика одна а символы разные.

## Примеры
**DSL-компилятор:** kwargs в определении функции (`fn f(x, cwd = ".")`) и в вызове (`f(x, cwd = root)`) и в config блоке (`let X = 42`) — везде `=`. Альтернатива с `:` для kwargs (Python-style `f(cwd=root)` vs Kotlin `f(cwd: root)`) была отвергнута: `=` везде где происходит присваивание.

**CSS vs JS:** `color: red` (CSS) и `color = 'red'` (JS) — разные символы для одного действия. Пользователь переключающийся между ними постоянно ошибается.

## Следствия
- Язык учится быстрее: одно правило вместо таблицы исключений.
- Ошибки типа "перепутал `:` и `=`" исчезают по конструкции.
- Парсер проще: нет контекстно-зависимой интерпретации одного символа.
