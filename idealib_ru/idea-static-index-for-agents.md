## Формулировка
Если в репозитории есть структурированный контент который агент или LLM должен уметь навигировать — положи в корень статический INDEX.md со списком всех файлов. Один fetch вместо ls которого может не быть.

## Обоснование
Агенты и LLM работают с репозиторием через HTTP (raw.githubusercontent.com) или API — у них нет гарантированного доступа к `ls`, `find`, или directory listing. Статический индекс решает это без инфраструктуры: один файл по предсказуемому URL содержит полную карту контента с метаданными.

Устаревание решается git hook'ом — индекс регенерируется автоматически при каждом пуше. Связь с → committed-git-hooks.

Граница применимости: оправдано когда контент структурированный и его больше 10-15 единиц. Для маленького репо достаточно README.

## Примеры
**name-police/INDEX.md:** список всех idea-файлов из idealib_en/, idealib_ru/, idealib_attic/ с id, strength и тегами. Агент в новой сессии: один fetch INDEX.md → полная картина → точечный fetch нужных юнитов. Никакого ls, никаких сюрпризов.

**Паттерн записи:**
```markdown
- [idea-slug](path/to/file.md) `strength` [tags]
```

## Следствия
- URL индекса предсказуем: `https://raw.githubusercontent.com/user/repo/main/INDEX.md` — можно хардкодить в промпты.
- Индекс полезен не только агентам — человек тоже ориентируется быстрее по одному файлу чем по дереву папок.
- Связь с → knowledge-colocation: индекс живёт в репо, не протухает, не требует внешнего сервиса.
- Генератор индекса (`scripts/gen-index.sh`) — отдельный файл, его можно копировать в другие репо с той же структурой.
