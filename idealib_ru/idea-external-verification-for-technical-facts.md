## Формулировка
Для верификации конкретного технического факта о внешней системе (API, поведение библиотеки, runtime) — запрос к независимому источнику ценнее консенсуса внутри текущего диалога.

## Обоснование
Внутри диалога участники могут сходиться на неверном ответе — потому что никто не проверял, а уверенность создаётся повторением. LLM особенно подвержен этому: может галлюцинировать API с высокой уверенностью, особенно для малоизвестных систем. Независимый источник (другая модель, документация, эксперимент) разрывает этот круг.

Граница применимости: для фактов которые можно проверить эмпирически или через авторитетный источник. Не применимо к вопросам где нет объективного ответа.

## Примеры
**DSL-компилятор:** вопрос о `FindPrev` в micro Lua API. Внутри диалога была неуверенность — я написал предположительный код. Привлечение Грока дало конкретный пример с реальным кодом и двумя альтернативами (plain search и PCRE через luare). Это изменило реализацию: выбрали итерацию по строкам без внешних зависимостей.

**Общий паттерн:** "Грок говорит вот так" / "в документации написано X" — сигнал для обновления позиции, не для защиты текущей.

## Следствия
- Разные модели имеют разные пробелы в знаниях — комбинирование снижает вероятность галлюцинации.
- Экспертиза в диалоге не должна быть монополией одного источника.
- Для LLM-assisted разработки: держать доступ к нескольким моделям и использовать их как независимых рецензентов технических деталей.
