## Формулировка
`github.com/blob/` URL — это человеческий интерфейс (HTML-страница); `raw.githubusercontent.com` — машинный контракт. При проектировании index-файлов для агентов формат ссылок — это API-решение, не косметика.

## Обоснование
Агенты и LLM фетчат содержимое файлов, а не отрендеренные страницы. Blob URL возвращает HTML-обёртку с контентом внутри — нефетчабельна без браузера. Raw URL возвращает файл напрямую по HTTP. Для публичных репо это работает без аутентификации; для приватных требует `Authorization: Bearer TOKEN` в заголовке запроса. Различие важно на этапе проектирования: index-файл с blob-ссылками сломан для любого небраузерного потребителя — независимо от того, насколько хорошо структурирован остальной индекс.

## Примеры
Raw URL для публичного репо: `https://raw.githubusercontent.com/owner/repo/main/path/to/file.md` — фетчится curl, fetch(), tool call LLM.

Тот же файл как blob URL: `https://github.com/owner/repo/blob/main/path/to/file.md` — возвращает HTML, бесполезен для агентов.

## Следствия
- Любой статический индекс для агентного потребления должен использовать исключительно raw URL.
- Для приватных репо стратегия передачи токена (Authorization-заголовок) решается на уровне агента, не кодируется в индексе — индекс остаётся без токена.
- Баги формата URL в генераторах индексов молчаливые: индекс выглядит корректно для человека, но каждая ссылка сломана для машины.
