## Формулировка
В каждом проекте должен быть файл "чему мы научились пока строили это" — набор деконтекстуализированных технических и концептуальных инсайтов, извлечённых из реального опыта постройки именно этого проекта.

## Обоснование
Знание рождается в процессе — пока решаешь проблему, находишь паттерны и антипаттерны которые нигде не записаны. Если их не зафиксировать сразу, они испаряются: следующий разработчик (или ты сам через полгода, или LLM в новой сессии) наступает на те же грабли.

Отличие от обычного README: README объясняет *что* делает проект. Skills-файл объясняет *что мы узнали пока его делали* — это ортогональное знание. Отличие от комментариев в коде: инсайты часто надпроектные — применимы в других проектах, в других технологиях.

Связь с → knowledge-unit-store: skills-файл это сырой материал для idealib. Лучшие инсайты из него заслуживают быть вынесены в отдельные атомарные юниты с тегами.

## Примеры
**name-police/SKILLS.md:** содержит 12 паттернов — от "используй @babel/parser вместо acorn" до "whitelist эффективнее blacklist для запрета контекстов". Каждый паттерн самодостаточен и применим за пределами проекта.

**Структура записи в skills-файле:**
```
## N. Название паттерна
Одно предложение — суть.
Код или пример если нужен.
Почему это важно / где ещё применимо.
```

## Следствия
- LLM в новой сессии, получив skills-файл проекта, мгновенно восстанавливает контекст решений — не надо объяснять "почему мы используем X а не Y". Связь с → claude-github-workflow: такой файл особенно ценен именно потому что LLM теряет память между сессиями.
- Новый разработчик за 10 минут понимает нетривиальные решения которые иначе узнаёт через месяц проб и ошибок.
- Skills-файл пишется *во время* работы, не после — инсайт фиксируется пока он свежий. Связь с → asymmetric-cost-of-omission: стоимость незаписанного инсайта невосполнима.
- Со временем паттерны повторяющиеся в skills-файлах разных проектов — кандидаты в idealib как core-юниты.
