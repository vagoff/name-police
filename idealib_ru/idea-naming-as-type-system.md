## Формулировка
Достаточно жёсткая конвенция именования делает анализ потоков данных сводимым к синтаксической проверке — без type inference, без графа потоков, только имена.

## Обоснование
Если контролировать три вещи одновременно: (1) только `new C()` может породить имя паттерна `xxx*`, (2) `xxx*` имена текут только в другие `xxx*` имена, (3) `xxx*` имена появляются только в разрешённых синтаксических позициях — то система замкнута. Вход в `xxx*`-пространство только через `new C()`, выход невозможен. Инвариант не выводится, а постулируется синтаксически. Это и есть эквивалентность.

Граница применимости: работает только для именованных классов-инстансов. Примитивы неразрешимы — там система не применима и не нужна.

## Примеры
**name-police:** `dbt` — хорошее имя для DB_ToolBox. Линтер гарантирует что любая переменная `dbt*` содержит именно DB_ToolBox — без TypeScript, без аннотаций.

**Кросс-доменно:** Hungarian notation делала то же самое вручную в 80-х. Разница: name-police проверяет машинно, и использует хорошие имена вместо зашумлённых префиксов.

## Следствия
- Grep по паттерну имени даёт полную и точную картину использования класса — без сюрпризов.
- Разные разработчики не могут называть один инстанс `helper`, `obj`, `tmp` — система не позволит.
- Область где работает — покрывается на 100% с нулём false positives. Лучше покрыть малую область идеально, чем большую приближённо.
