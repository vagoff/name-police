## Формулировка
Git hooks должны жить в репозитории как обычные файлы, а не в `.git/hooks/`. Используй `core.hooksPath .githooks` — тогда хуки версионируются, передаются при клоне и не теряются.

## Обоснование
`.git/hooks/` не версионируется и не клонируется — хук написанный одним разработчиком невидим остальным и исчезает при повторном клоне. Это антипаттерн той же природы что документация в Notion: работает локально, теряется при передаче. `core.hooksPath` позволяет хранить хуки в любой папке репозитория — они становятся обычными файлами с историей изменений.

Граница применимости: требует одной команды после клона. Можно автоматизировать через `Makefile` или `package.json postinstall`.

## Примеры
**Установка:**
```bash
# один раз после клона
git config core.hooksPath .githooks
```

**name-police:** `.githooks/pre-push` автоматически регенерирует `INDEX.md` и амендит последний коммит если файл изменился. Хук в репо — значит работает у всех, версионируется, виден в `git log`.

## Следствия
- Хук это часть проекта — он должен жить рядом с кодом. Связь с → knowledge-colocation.
- `git log .githooks/` показывает историю изменений хуков как обычных файлов.
- Новый разработчик видит хуки в репо, читает что они делают, устанавливает одной командой.
- Автоматизация которая живёт только на машине автора — не автоматизация, а личный скрипт.
