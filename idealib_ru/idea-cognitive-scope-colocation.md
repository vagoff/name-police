## Формулировка
Модули которые попадают в scope внимания вместе — должны лежать в одной папке. Критерий организации не архитектурный слой и не тип, а минимизация когнитивного трения как интеграла по всем сессиям работы.

## Обоснование
Архитектурные слои описывают природу модуля. Когнитивный scope описывает паттерн использования. Это ортогональные вещи — и для навигации важнее второе. Модуль который структурно принадлежит "infra" но всегда открывается вместе с "domain" — создаёт трение при каждой сессии работы.

Правильная папка та из которой редко нужно выходить за соседним файлом в рамках одной задачи. Неправильная папка обнаруживается эмпирически: если постоянно открываешь файлы из двух папок одновременно — они должны быть одной.

Граница применимости: принцип работает когда паттерны использования устоялись. В новом проекте без истории — начинай с архитектурных слоёв, потом корректируй по реальному трению.

## Примеры
**Код:** модуль валидации который всегда открывается вместе с моделью — лежит рядом с моделью, не в отдельной `validation/` папке. Модуль который соединяет два слоя и не принадлежит ни одному — лежит там где его чаще всего используют.

**Антипример:** `utils/`, `helpers/`, `common/` — папки организованные по типу а не по scope. Максимальное трение: чтобы понять что внутри нужно открыть каждый файл.

## Следствия
- Модули "между слоями" перестают быть проблемой — они просто ложатся по паттерну использования.
- Папочная структура становится живой: если паттерны работы меняются — структура должна меняться вслед.
- Запрет `utils/` выводится из этого принципа: utils не имеет когнитивного scope — это свалка а не папка.
- Применимо не только к коду: любая организация файлов, документов, заметок подчиняется тому же принципу.
