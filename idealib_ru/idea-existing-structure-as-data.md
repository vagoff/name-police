## Формулировка
Прежде чем вводить новый тип данных — проверь, не является ли нужная структура уже существующим примитивом системы в другом виде. Введение нового типа стоит дороже чем кажется.

## Обоснование
Новый тип тянет за собой: синтаксис литерала, операции (индексирование, длина, итерация), сериализацию в state, передачу между функциями. Это целый пласт языка или API. Если существующий примитив покрывает задачу — нет смысла платить эту цену.

Граница применимости: не работает когда существующий примитив семантически неверен (хранить список в строке через разделитель — технически работает, семантически ужасно). Подходит только когда существующая структура является естественным представлением данных, а не просто техническим хаком.

## Примеры
**DSL-компилятор:** для навигации по результатам grep нужен был список строк с текущей позицией. Вместо введения типа `list` — использовали текстовый буфер редактора. Буфер с результатами grep уже является списком строк. `buf.find_next_re_from(gbuf, pos, pattern)` итерирует строки буфера. Паттерн для поиска — конфигурируемый regex. Не пришлось добавлять синтаксис массивов.

**Unix:** текстовый поток как универсальная структура данных между инструментами. `grep | head | sort` — каждый инструмент работает с тем же примитивом.

## Следствия
- Язык остаётся меньше: меньше синтаксиса, меньше когнитивной нагрузки.
- Все инструменты работающие с существующим примитивом автоматически применимы к новой задаче.
- В данном случае: quickfix-функциональность получила все текстовые операции буфера бесплатно — поиск, навигация, отображение.
