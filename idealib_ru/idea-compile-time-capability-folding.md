## Формулировка
Compile-time константа известная на момент компиляции позволяет выполнить dead code elimination до генерации output — не оставляя в сгенерированном коде ни условия, ни мёртвых веток. Платформенные различия исчезают из output, хотя написаны один раз в source.

## Обоснование
Альтернатива — runtime проверка возможностей (`if (hasFeature('x'))`) — оставляет в output весь код обеих веток. Это мусор: неиспользуемая ветка занимает место, может содержать недостижимые ошибки, усложняет читаемость. Если возможность платформы известна статически — она должна быть разрешена статически.

Механика: `command_exists("x")` компилируется в строку `"true"` или `"false"`. `emit_if` проверяет является ли условие константой и если да — эмитирует только нужную ветку.

Граница применимости: работает только для различий известных до компиляции. Runtime-зависимые условия (наличие файла, состояние системы) требуют runtime проверки.

## Примеры
**DSL-компилятор:** для Neovim `editor.command_exists("find_in_files")` → `true`, вся else-ветка с fallback через `rg` не попадает в Lua. Для micro → `false`, native-ветка не попадает. Код написан один раз, output — только то что нужно платформе.

**C preprocessor:** `#ifdef TARGET_ARM` — та же идея, другой синтаксис. Compile-time ветвление по платформе. DSL-вариант чище: нет ifdef-ов в source, условие написано как обычный if.

**Tree shaking в бандлерах:** статический анализ импортов, неиспользуемый код не попадает в bundle — тот же принцип.

## Следствия
- Сгенерированный код идиоматичен для платформы: нет артефактов multi-platform source.
- Добавление новой платформы не требует изменения DSL-плагинов — только объявления capabilities в генераторе.
- `shell_ext` добавился за 10 минут — та же механика, другой capabilities Set.
