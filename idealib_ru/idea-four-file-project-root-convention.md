## Утверждение
Корень проекта с четырьмя документами — README.md, ARCHITECTURE.md, CODEBASE.md,
RESEARCH.md — покрывает все вопросы которые задаёт разработчик или агент при входе
в кодовую базу, без пересечений между файлами.

## Обоснование
Каждый файл отвечает на свой вопрос:

| Файл | Вопрос |
|---|---|
| README.md | Что это и как установить? |
| ARCHITECTURE.md | Почему так устроено? |
| CODEBASE.md | Как навигировать и запускать? |
| RESEARCH.md | Что изучали до принятия решений? |

Граница между ARCHITECTURE и RESEARCH — это точка принятия решения. ARCHITECTURE
фиксирует что было решено и почему. RESEARCH фиксирует состояние знания до решения
— обзор рынка, оценённые альтернативы, рассуждение о границах скоупа. Без RESEARCH
у ARCHITECTURE нет контекста. Без ARCHITECTURE у RESEARCH нет выводов.

README — для людей подходящих к проекту впервые. CODEBASE — для агентов и
разработчиков уже внутри проекта. ARCHITECTURE — для тех кто спрашивает «почему».
RESEARCH — для тех кто спрашивает «что ещё рассматривалось».

Граница применимости: применимо к проектам с нетривиальными архитектурными
решениями и предрешенческим исследованием. Небольшому скрипту с очевидным дизайном
достаточно одного README.

## Примеры
**name-police:** README (установка, использование, примеры), ARCHITECTURE (двухпроходный
дизайн, почему не Semgrep/ESLint, инвариант замкнутой системы), CODEBASE (grep-паттерны,
поиск по индексу, как расширять), RESEARCH (выбор инструмента для lex-police, отчёты
Grok x2, сравнительная таблица, решения о границах скоупа).

## Следствия
- Агент клонирующий репо читает четыре файла и получает полную ориентацию: что,
  почему, как, и какие альтернативы были отвергнуты — до начала сессии.
- Ни один документ не становится свалкой — у каждого есть определённая ответственность.
- Предрешенческое знание переживает границы сессий (критично для AI-assisted разработки).
- Конвенция переносимая: любой проект может её принять без модификации.
- Связано: → pre-decision-research-as-artifact, → rejection-log-as-design-doc,
  → knowledge-colocation.
