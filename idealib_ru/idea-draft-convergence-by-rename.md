## Формулировка
Чтобы свести несколько нумерованных черновиков в один канонический файл без потери истории: сначала переименовать самый ранний черновик в целевой slug, затем перезаписать его финальным содержимым. Никогда не делать оба шага в одном коммите.

## Обоснование
Это механическая реализация → numbered-draft-resolution. Ключевое — последовательность: сначала установить идентичность (rename), потом развить содержимое. Определение rename в git работает на диффе между двумя состояниями коммитов — изоляция переименования максимизирует достоверность определения.

Коммит переименования имеет нулевое или минимальное изменение содержимого: git вычисляет схожесть как (совпадающие строки) / (всего строк). Чистое переименование даёт ~100%. Добавление изменений содержимого снижает этот показатель и может опустить его ниже порога git по умолчанию в 50%, из-за чего история выглядит как delete+create.

## Примеры
```
# коммит 1 — только rename
git mv idea-xxx1.md idea-xxx.md
git commit -m "rename: idea-xxx1 → idea-xxx"

# коммит 2 — только content
cp idea-xxx2.md idea-xxx.md
git rm idea-xxx2.md
git add idea-xxx.md
git commit -m "update: idea-xxx — merge extended draft, remove xxx2"
```

## Следствия
- `git log --follow idea-xxx.md` прослеживается сквозь переименование до создания исходного `xxx1`.
- Дифф в коммите 2 показывает только дельту содержимого между двумя черновиками — чисто и читаемо.
- Обобщается на любое слияние файлов, не только на черновики идей.
