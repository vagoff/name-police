## Формулировка
Хорошие имена уже несут типовую информацию — machine enforcement делает её формальной без добавления шума.

## Обоснование
Hungarian notation пыталась закодировать тип в имя через префикс — это добавляло когнитивную нагрузку. name-police наоборот: `dbt` это уже хорошее имя для DB_ToolBox, оно короткое, читаемое, очевидное. Линтер не добавляет церемоний — он формализует то что хороший ревьюер и так бы потребовал.

Граница применимости: требует что в домене уже есть устоявшиеся хорошие имена. Если домен новый и имён нет — сначала нужна конвенция, потом enforcement.

## Примеры
**name-police:** `dbt`, `new_dbt`, `dbt2` — это естественные имена для переменных содержащих DB_ToolBox. Паттерн `^(dbt|new_dbt|dbt2|tmp_dbt)$` это просто их перечисление.

**Unix:** `fd` для file descriptor, `pid` для process id — устоявшиеся хорошие имена. Если бы был линтер — паттерн `^fd[0-9]*$` был бы естественным.

## Следствия
- Разные разработчики называющие один класс `helper`, `obj`, `myTool` — это сигнал плохих имён, не только нарушение конвенции.
- Паттерн имён в конфиге name-police — это документация доменных конвенций, машинно проверяемая.
- Enforcement без хороших имён — бессмысленен. Хорошие имена без enforcement — теряются при росте команды.
