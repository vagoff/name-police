## Формулировка
Когда логика генерации правил для инструмента сложнее самих правил — пиши checker напрямую, инструмент стал промежуточным слоем без ценности.

## Обоснование
Semgrep оправдан для простых паттернов написанных вручную. Как только появляется скрипт генерации YAML-правил по сигнатурам — ты фактически пишешь анализатор, только с неудобным промежуточным форматом. Прямой AST walker убирает этот слой и даёт полный контроль.

Граница применимости: Semgrep быстрее для простых случаев (5-10 правил, написанных вручную). Кастомный walker оправдан когда: нужен cross-file анализ, нужна whitelist-логика, правила генерируются программно.

## Примеры
**name-police:** начали с вопроса "можно ли на semgrep?" Ответ: можно п.1 и п.2, п.3 и п.4 — нет. Генератор правил для п.3 по сложности = сам checker. Решение: @babel/parser + traverse.

**Компиляторные плагины:** Babel plugin проще чем codemod на jscodeshift для сложных трансформаций — меньше абстракций между тобой и AST.

## Следствия
- Точка перегиба: "нужен скрипт для генерации правил инструмента" → пора писать checker.
- ESLint plugin как тонкая обёртка над custom checker — правильный компромисс: ядро без зависимости от ESLint, интеграция с редакторами бесплатно.
- Semgrep остаётся полезным для ad-hoc проверок и security-паттернов — не универсальное решение.
