## Формулировка
Для статического анализа требующего глобальной информации: pass1 собирает всё по всем файлам, pass2 проверяет каждый файл используя накопленное.

## Обоснование
Проверка call-site требует знания сигнатуры функции которая может быть определена в другом файле. Однопроходный анализ вынужден либо откладывать проверки (сложно), либо работать только в пределах файла (ограниченно). Два прохода разделяют сбор информации и её использование — классическое разделение ответственности.

Граница применимости: файлы парсятся дважды — приемлемо для линтера, неприемлемо для hot path. Для очень больших кодовых баз pass1 можно кешировать.

## Примеры
**name-police:** pass1 строит `Map<methodName, paramNames[]>` по всем файлам. pass2 использует эту карту при проверке каждого вызова.

**Компиляторы:** большинство компиляторов многопроходные — первый проход собирает символы, следующие используют таблицу символов.

## Следствия
- Pass1 и pass2 независимы — можно тестировать отдельно.
- Накопленное состояние pass1 передаётся в pass2 как обычный объект — никакого глобального состояния.
- ESLint plugin может запускать только pass2 (с ограниченным pass1 по одному файлу) — полная проверка только в CLI.
